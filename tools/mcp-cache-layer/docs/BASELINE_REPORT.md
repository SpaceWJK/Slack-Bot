# MCP Cache Layer — 병목 계측 리포트 (Baseline)

> 분석 일시: 2026-03-10
> 데이터 기간: 2026-03-09 ~ 2026-03-10
> 데이터 소스: `logs/wiki_query.log` (4건), `logs/gdi_query.log` (16건)

## 1. 데이터 한계

/wiki와 /gdi 기능은 2026-03-09 배포 직후라 로그가 20건뿐.
통계적 유의성은 낮지만, MCP 호출 비용의 규모감과 캐시 적용 패턴은 확인 가능.

## 2. Wiki MCP 분석 (4건)

| 지표 | 값 |
|------|-----|
| 총 쿼리 수 | 4 |
| 평균 응답시간 | 4,429ms |
| ask_claude 평균 | 5,632ms |
| get_page 평균 | 817ms |
| 에러율 | 0% |

### 동일 쿼리 재조회

| 쿼리 | 횟수 | 시간 범위 |
|------|------|----------|
| "서비스 장애 리포트" + AI질의 | 3회 | 17:54 → 18:00 → 18:39 (45분간) |

**재조회 비율: 75% (4건 중 3건이 동일 쿼리)**

→ 캐시 효과 극대화 가능. 첫 1회만 MCP 호출하면 나머지 2회는 캐시 hit.

### 비용 분해

```
ask_claude 1회 비용:
  MCP cql_search .... ~500ms
  MCP get_page ...... ~300ms  (본문 조회)
  Claude API 호출 ... ~4800ms (본문 → AI 답변)
  합계 .............. ~5600ms

캐시 적용 시:
  캐시 hit .......... ~10ms
  Claude API 호출 ... ~4800ms (본문 → AI 답변, 캐시된 본문 사용)
  합계 .............. ~4810ms
  MCP 절감 .......... ~800ms (14% 시간 절감)
```

**주의**: Wiki ask_claude에서 시간의 85%는 Claude API 호출. MCP 캐시로 줄일 수 있는 것은 15%.
단, **토큰 절감**이 진짜 가치 — MCP 호출 결과가 컨텍스트에 삽입되지 않으므로 API 비용 감소.

## 4. 종합 판단

### 캐시 효과 예측

| 지표 | Wiki | GDI | 합산 |
|------|------|-----|------|
| 총 쿼리 | 4 | 16 | 20 |
| 캐시 hit 가능 | 3 (75%) | 11 (69%) | 14 (70%) |
| 절감 MCP 호출 | 3회 | 11회 | 14회 |
| 절감 시간 (추정) | ~16s | ~62s | ~78s |

### 결론

- **재조회 비율 70%** → 캐시 적용 충분히 유효 (판단 기준 20% 초과)
- **GDI가 더 큰 효과**: 쿼리 수 多 + 응답시간 長 + 반복률 高
- **우선순위 수정 제안**: Phase 1에서 Wiki 먼저 하되, GDI도 빠르게 Phase 2 진행 권장

### 추적 필요 지표 (향후)

캐시 적용 후 아래를 `wiki_query.log`/`gdi_query.log`에 추가 기록하여 효과 측정:
- `cache_status`: hit / miss / stale / disabled
- `cache_hit_time_ms`: 캐시 hit 시 응답 시간 (MCP 호출 없는 경우)
